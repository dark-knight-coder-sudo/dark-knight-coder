01 Google

Senior DELive coding · 45 min

Google's own browser editor — no execution, no autocomplete. One SQL question with a follow-up chain, then one Python question. The interviewer probes time/space complexity out loud the entire time. Loop: ① SQL + Python coding → ② ETL design + ML → ③ hiring manager.

SQL — year-over-year revenue change

Given data.revenue:

customeryearrevenue
ABC2018100
ABC20191000
ABC2020200
ABC2021300
BCD201811000
BCD202072000
BCD2021300
Calculate the year-over-year % change in revenue for each customer as a new column.
SELECT customer, year, revenue,
  (revenue - LAG(revenue) OVER (PARTITION BY customer ORDER BY year)) * 100.0
    / LAG(revenue) OVER (PARTITION BY customer ORDER BY year) AS yoy_pct_change
FROM data.revenue;
Follow-up — BCD is missing 2019. Find the missing years per customer.

Generate the full year range per customer (min→max), then anti-join the actual rows:

WITH bounds AS (
  SELECT customer, MIN(year) min_y, MAX(year) max_y FROM data.revenue GROUP BY customer
),
expected AS (
  SELECT b.customer, y.year
  FROM bounds b JOIN years y ON y.year BETWEEN b.min_y AND b.max_y  -- generated series
)
SELECT e.customer, e.year
FROM expected e
LEFT JOIN data.revenue r ON r.customer = e.customer AND r.year = e.year
WHERE r.year IS NULL;
Follow-up — flag significant outlier changes in the new column, e.g. 2× over or under the customer's average change.
WITH yoy AS ( /* query above */ )
SELECT customer, pct_change,
       CASE WHEN ABS(pct_change) > 2 * AVG(ABS(pct_change))
                 OVER (PARTITION BY customer)
            THEN 'outlier' END AS flag
FROM yoy;

The exact threshold matters less than the reasoning: define a per-customer baseline, then flag deviations beyond a multiple of it.

Python — anagram check

Write a function that determines if two given strings are anagrams. Propose and develop different approaches, and explain their time and space complexities. (Examples: tar/rat, listen/silent.)
# Approach 1 — sort and compare · O(n log n) time, O(n) space
def is_anagram(a, b):
    return sorted(a) == sorted(b)

# Approach 2 — counter · O(n) time, O(k) space
from collections import Counter
def is_anagram(a, b):
    return len(a) == len(b) and Counter(a) == Counter(b)

They want you to volunteer the trade-off — sorting is simpler, counting is asymptotically better and short-circuits on length — not wait to be asked. The round is won or lost on the follow-up chain and how calmly you extend your first answer.

03 HCSC

Senior MLELive coding

Live coding over video call in a shared editor — SQL first, then Python. LeetCode-easy/medium difficulty, but they watch how you structure and explain. Narrate as you go: silence while typing is the failure mode, not wrong syntax.

SQL — salary above department average

Find the names of employees whose salary is higher than the average salary of departments with more than 3 employees.
employees(emp_id, name, department_id, salary)
departments(department_id, dept_name)

WITH dept_stats AS (
  SELECT department_id, AVG(salary) avg_salary
  FROM employees
  GROUP BY department_id
  HAVING COUNT(*) > 3
)
SELECT e.name
FROM employees e
JOIN dept_stats d ON e.department_id = d.department_id
WHERE e.salary > d.avg_salary;

Probed after: why HAVING not WHERE, and the correlated-subquery alternative vs the CTE.

Python — valid parentheses

Given a string containing only ( ) { } [ ], determine if it's valid: brackets closed by the same type, in the correct order, every closer has an opener.
def is_valid(s):
    pairs = {')': '(', '}': '{', ']': '['}
    stack = []
    for ch in s:
        if ch in pairs:
            if not stack or stack.pop() != pairs[ch]:
                return False
        else:
            stack.append(ch)
    return not stack

O(n) time, O(n) space. Say the invariant out loud: the stack always holds the currently-unclosed openers, innermost on top.

Python — flatten nested JSON

Flatten the given JSON — nested keys join with underscores: contact.phone.home → contact_phone_home.
def flatten(d, parent="", sep="_"):
    out = {}
    for k, v in d.items():
        key = f"{parent}{sep}{k}" if parent else k
        if isinstance(v, dict):
            out.update(flatten(v, key, sep))
        else:
            out[key] = v
    return out

The most DE-flavored Python question there is — it shows up everywhere. Know the recursive version cold, and expect the follow-up: what about lists inside the JSON? (Index into the key: contact_phones_0.)

04 Kohl's

Data EngineerLive coding · Python

Plain Python, stdlib only — pandas/numpy explicitly banned, collections/datetime encouraged.

A flat list holds purchase records — every 4 elements are one transaction: SKU, item description, price ('$10.00'), purchase date ('YYYY-MM-DD'). Print one formatted string: "{most_common_sku} {average_price} {min_date} {max_date}" — average price of the most common SKU to 2 decimals, plus that SKU's first and last purchase dates.
purchases = [
    '123456', 'Nike T-shirt', '$12.65', '2026-01-01',
    '194850', 'Vans Shoes',   '$89.99', '2026-01-04',
    # … 10 rows total
]

def aggregate_purchase_data(data):
    sku_info = {}
    for i in range(0, len(data), 4):
        sku   = data[i]
        price = float(data[i + 2].replace('$', ''))
        date  = data[i + 3]
        info = sku_info.setdefault(sku,
            {"count": 0, "total": 0.0, "min_date": date, "max_date": date})
        info["count"] += 1
        info["total"] += price
        info["min_date"] = min(info["min_date"], date)
        info["max_date"] = max(info["max_date"], date)

    top  = max(sku_info, key=lambda s: sku_info[s]["count"])
    info = sku_info[top]
    return f"{top} {info['total']/info['count']:.2f} {info['min_date']} {info['max_date']}"

What they're testing: stride iteration over a flat list, string→number cleaning, and one-pass aggregation into a dict — GROUP BY in plain Python. The nice catch: ISO dates compare correctly as strings, so no datetime parsing needed — saying that out loud scores points. Useful drill: solve it in SQL first, then translate.

05 Andela

DE · Client screeningTake-home

Take-home with provided datasets, two parts. Graded on judgment, not code volume: how you handle nulls, dupes, and join mismatches — and whether you wrote the decisions down.

Part 1 — messy user records

A spreadsheet of synthetic user records (first_name, last_name, email, home_phone, mobile_phone, street_address, city, state, postal_code, date_of_birth), deliberately dirty: missing values in every column, inconsistent phone formats, near-duplicate people. They want a repeatable cleaning pipeline — standardize formats, define a dedupe key, decide explicitly what happens to unfixable rows, document every decision.

Part 2 — usage analytics

Two CSVs to join on username:

usage_data.csv: date, username, feature, sessions, time_spent, average_time_spent
user_data.csv:  first_name, last_name, username, role, region, subscription_plan

Landmines: subscription_plan has blanks, and the join key only exists cleanly on one side if Part 1 was done right. Typical asks: usage by feature/role/region, free-trial vs paid behavior, most and least engaged segments — presented as findings, not just tables. Keep it boring and reproducible: one runnable script or notebook plus a short README of assumptions.

07 Beacon Hill

Coding — Exponents of 2

Input: first line is T. Each of the next T lines is a number. Print 1 if it's a power of two, else 0.

The catch: inputs can be non-integers, and the intended definition accepts fractional powers (0.5, 0.25) since halving/doubling reaches exactly 1.0. Anything ≤ 0 is 0.

import sys
def is_power_of_two(n):
    if n <= 0: return False
    while n > 1: n /= 2
    while n < 1: n *= 2
    return n == 1.0

data = sys.stdin.read().split('\n')
t = int(data[0])
out = ['1' if is_power_of_two(float(data[i])) else '0' for i in range(1, t + 1)]
print('\n'.join(out))

For integers only, the one-liner is n > 0 and (n & (n - 1)) == 0 — worth saying out loud even if the float version is what the tests require.

Coding — Sorting Bank Accounts

Per test case: an integer N, then N account strings. Print each unique account with its count, sorted by account number. Blank line separates cases.
import sys
from collections import Counter

data = sys.stdin.read().split('\n')
idx = 0
t = int(data[idx].strip()); idx += 1
blocks = []
for _ in range(t):
    n = int(data[idx].strip()); idx += 1
    accounts = [data[idx + j].strip() for j in range(n)]
    idx += n
    if idx < len(data) and data[idx].strip() == '': idx += 1  # separator
    counts = Counter(accounts)
    blocks.append('\n'.join(f"{a} {counts[a]}" for a in sorted(counts)))
print('\n\n'.join(blocks))

What it tests: careful index-based stdin parsing with a variable number of lines per case, Counter for the tally, sorted() on the keys for ordering.

Verbal round — reported scenarios

In an ETL process, what's a common method for handling data quality issues during the transformation phase?

Validate on the way through and quarantine, don't drop: schema/type and constraint checks (not-null, ranges, referential) applied as data transforms; route failing rows to a dead-letter / reject table with the reason attached instead of silently discarding; standardize and de-dupe; emit DQ metrics (rows in vs out, % rejected, null rates) so regressions are visible. Tools named: dbt tests, Great Expectations, rule tables.

A 2-TB Delta table partitioned by event_date has slow reads and high cost. In 3 minutes, describe the exact sequence of Databricks actions to diagnose and fix it.
  1. Diagnose first — Spark UI / query profile, then DESCRIBE DETAIL for numFiles and size (→ avg file size) and partition skew. Slow partitioned reads are almost always the small-file problem.
  2. Compact — OPTIMIZE toward the ~128 MB–1 GB target; enable auto-optimize / optimizeWrite so it stays compacted.
  3. Z-ORDER on high-cardinality columns in your WHERE/join predicates (not the partition column) — that's what enables data-skipping.
  4. Reclaim safely — VACUUM old files but keep the default 7-day retention; never RETAIN 0 on a live table (breaks time-travel and concurrent readers).
  5. Compute — turn on Photon, right-size + autoscale the cluster, use Delta caching for hot reads.
  6. Query side — push event_date predicates for partition pruning, select only needed columns, broadcast small dimension joins.
  7. Rethink layout — if event_date makes tiny partitions, coarsen to month + Z-ORDER, or use liquid clustering.

The framing that scores: diagnose → compact → cluster → reclaim → tune compute → fix the query → rethink layout. Naming the file-size target and the VACUUM-retention safety note is what separates a senior answer from a keyword list.
