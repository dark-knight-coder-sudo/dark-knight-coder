Google Senior Data Engineer Interview: The Exact Questions From My Live Coding Round

Field notes from a real Google Senior Data Engineer loop (2026). This was round one — live coding — written down exactly as it was asked, with clean solutions. No 500-question dump, just what actually came up.

The format

About 45 minutes in Google's own browser editor — no execution, no autocomplete. One SQL question with a follow-up chain, then one Python question. The interviewer probes time and space complexity out loud the entire time. The full loop for this role: SQL + Python coding, then ETL design + ML, then hiring manager.

SQL: year-over-year revenue change

Given a table data.revenue:

customer | year | revenue
ABC      | 2018 | 100
ABC      | 2019 | 1000
ABC      | 2020 | 200
ABC      | 2021 | 300
BCD      | 2018 | 11000
BCD      | 2020 | 72000
BCD      | 2021 | 300

Question 1: Calculate the year-over-year % change in revenue for each customer as a new column.

SELECT customer, year, revenue,
  (revenue - LAG(revenue) OVER (PARTITION BY customer ORDER BY year)) * 100.0
    / LAG(revenue) OVER (PARTITION BY customer ORDER BY year) AS yoy_pct_change
FROM data.revenue;

Follow-up: BCD is missing 2019. Find the missing years per customer.

Generate the full year range per customer (min to max), then anti-join the actual rows:

WITH bounds AS (
  SELECT customer, MIN(year) min_y, MAX(year) max_y
  FROM data.revenue GROUP BY customer
),
expected AS (
  SELECT b.customer, y.year
  FROM bounds b
  JOIN years y ON y.year BETWEEN b.min_y AND b.max_y  -- generated series
)
SELECT e.customer, e.year
FROM expected e
LEFT JOIN data.revenue r
  ON r.customer = e.customer AND r.year = e.year
WHERE r.year IS NULL;

Follow-up: flag significant outlier changes in the new column — for example, 2x over or under the customer's average change.

WITH yoy AS ( /* query above */ )
SELECT customer, pct_change,
       CASE WHEN ABS(pct_change) > 2 * AVG(ABS(pct_change))
                 OVER (PARTITION BY customer)
            THEN 'outlier' END AS flag
FROM yoy;

The exact threshold matters less than the reasoning: define a per-customer baseline, then flag deviations beyond a multiple of it.

Python: anagram check

Write a function that determines if two given strings are anagrams. Propose and develop different approaches, and explain their time and space complexities. (Examples given: tar/rat, listen/silent.)

# Approach 1 — sort and compare · O(n log n) time, O(n) space
def is_anagram(a, b):
    return sorted(a) == sorted(b)

# Approach 2 — counter · O(n) time, O(k) space
from collections import Counter
def is_anagram(a, b):
    return len(a) == len(b) and Counter(a) == Counter(b)

They want you to volunteer the trade-off — sorting is simpler, counting is asymptotically better and short-circuits on length — not wait to be asked.

What actually decides this round

The SQL question is easy. The round is won or lost on the follow-up chain — missing years, outliers — and how calmly you extend your first query. And Big-O narration is its own skill: if saying "this is O(n log n) because of the sort, and here's how I'd get to O(n)" doesn't come out naturally while you type, drill it out loud before the interview. It sinks people who can otherwise solve the problem.
