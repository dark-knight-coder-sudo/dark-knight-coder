Healthcare Data Engineering Interviews: Real Questions From Providence and HCSC

Field notes from two real healthcare data engineering loops (2026) — a Senior DE role at Providence and a Senior ML Engineer role at HCSC. Written down as asked, with clean solutions. Healthcare loops are practical — SQL, warehousing, pipelines — not algorithm puzzles, and they're winnable on fundamentals.

Providence — Senior Data Engineer

Azure Data Factory — enough to hold the conversation

  • The four objects: pipeline → activity → dataset → linked service. Be able to walk one flow through all four.
  • Integration Runtime: Azure IR (cloud) vs self-hosted IR. Healthcare loves self-hosted because clinical sources sit behind firewalls.
  • Triggers: schedule vs tumbling window — the distinction interviewers reach for. Tumbling windows are stateful, non-overlapping, and support backfill and dependencies.
  • The senior-sounding answer: metadata-driven pipelines — one parameterized pipeline plus a control table, instead of 200 copy-pasted pipelines.
  • Incremental loads: watermark column + lookup activity, idempotent re-runs.
  • Security (healthcare will care): managed identity over connection strings, Key Vault for secrets, private endpoints.
  • Comparisons to have ready: ADF vs Synapse pipelines, ADF vs Airflow, ADF vs Databricks workflows.

Providence live coding — the actual questions (EMR / lab / genomics)

This is what was really asked in the coding round, on real healthcare schemas. The SQL half uses two tables: ORDER_RESULT (one row per lab result — patientid, encounter_id, order_id, result_time, external_name, ord_num_value) and PATIENT_ENCOUNTERS (patientid, encounter_id, encounter_type, contact_date).

1.1 — Display the unique lab test names.

SELECT DISTINCT external_name FROM ORDER_RESULT;

1.2 — Average WBC result per patient.

SELECT patientid, AVG(ord_num_value) AS avg_wbc
FROM ORDER_RESULT
WHERE external_name = 'WBC'
GROUP BY patientid
ORDER BY patientid;

1.3 — Identify patients with Pancytopenia (abnormally low red cells, white cells, and platelets, all at once): Hgb/Hemoglobin < 8.0, Platelet Count < 50, WBC ≤ 2.0.

The pattern: pivot lab rows into columns with conditional aggregation, then require all three in HAVING so they co-occur for the same patient and encounter.

SELECT patientid, encounter_id
FROM ORDER_RESULT
GROUP BY patientid, encounter_id
HAVING MAX(CASE WHEN external_name IN ('Hgb','Hemoglobin') THEN ord_num_value END) < 8.0
   AND MAX(CASE WHEN external_name = 'Platelet Count'       THEN ord_num_value END) < 50
   AND MAX(CASE WHEN external_name = 'WBC'                  THEN ord_num_value END) <= 2.0
ORDER BY patientid;

1.4 — Same-day readmissions. From PATIENT_ENCOUNTERS (one row per encounter), find patients with multiple inpatient admissions on the same day.

SELECT patientid, DATE(contact_date) AS admit_day, COUNT(*) AS admissions
FROM PATIENT_ENCOUNTERS
WHERE encounter_type = 'Inpatient'
GROUP BY patientid, DATE(contact_date)
HAVING COUNT(*) > 1;

1.5 — All labs for one patient's inpatient encounter on a date. For patientid = 'CBC1301104124', find all lab orders during the inpatient encounter on 2020-10-20.

SELECT o.patientid, o.order_id, o.encounter_id, o.result_time,
       o.external_name, o.ord_num_value
FROM ORDER_RESULT o
JOIN PATIENT_ENCOUNTERS e ON o.encounter_id = e.encounter_id
WHERE o.patientid = 'CBC1301104124'
  AND e.encounter_type = 'Inpatient'
  AND DATE(e.contact_date) = '2020-10-20';

Biomarker data mart — the design and ops questions

The BIOMARKER table holds gene-sequencing order information. These are talk-through-it questions, not SQL:

  • A customer says they see no genomic data from a given lab (Caris) in the last two weeks — confirm it. Count orders per day per lab over the trailing ~60 days and show the last 14 are zero, or far below that lab's normal daily volume. Confirm it's lab-specific (others still flowing) and check raw ingestion counts against the mart to separate a source-side gap from a pipeline bug. One GROUP BY lab, order_date with a baseline comparison settles it.
  • How do you communicate the issue? Acknowledge scope and impact to the customer, open an incident, notify the vendor contact and downstream consumers, give an ETA and a workaround, then follow up with root cause and prevention. The point they probe: structured, proactive comms — not silence until it's fixed.
  • Add six more labs to the pipeline — design a scalable architecture and give a backlog with dependencies. Go metadata-driven (config per lab), not six copy-pasted pipelines: one parameterized ingestion framework, a config table per lab, a normalization layer to a canonical biomarker schema, and per-lab data-quality checks. Backlog: (1) inventory the six labs' formats and delivery methods, which blocks everything; (2) design the canonical schema and mapping config; (3) build the parameterized ingestion and normalization framework; (4) onboard labs one at a time via config; (5) per-lab DQ rules and volume-drop alerting — which makes the first question automatically detectable; (6) backfill, reconciliation, and cutover.

Providence Python round — free-text clinical NLP

Three questions on a patients_nlp.json file, where each patient record has a free-text note field.

Q1 — Parse the JSON and find NSCLC (Non-Small Cell Lung Cancer) patients from the note text. The subtlety that makes this more than a substring search: "small cell lung cancer" is contained inside "non-small cell lung cancer", so you check the NSCLC pattern first and require the non- prefix or the NSCLC acronym.

import json, re

with open('patients_nlp.json') as f:
    patients = json.load(f)

NSCLC = re.compile(r'\bNSCLC\b|non[-\s]?small[-\s]cell\s+lung', re.I)
SCLC  = re.compile(r'\bSCLC\b|small[-\s]cell\s+lung', re.I)

def classify(note):
    if NSCLC.search(note):
        return 'NSCLC'
    if SCLC.search(note):     # reached only if not NSCLC — "non-small" already excluded
        return 'SCLC'
    return None

nsclc = [p for p in patients if classify(p['note']) == 'NSCLC']
print([p.get('patient_id', p) for p in nsclc])

Q2 — Separate NSCLC and SCLC patients into two groups.

groups = {'NSCLC': [], 'SCLC': []}
for p in patients:
    label = classify(p['note'])
    if label:
        groups[label].append(p)

Q3 — The note also holds vitals. Extract resting heart rate and find the mean resting heart rate of NSCLC patients.

HR = re.compile(r'resting\s+heart\s+rate[:\s]*?(\d+)', re.I)

rates = []
for p in groups['NSCLC']:
    m = HR.search(p['note'])
    if m:
        rates.append(int(m.group(1)))

mean_hr = sum(rates) / len(rates) if rates else 0
print(f"Mean resting HR (NSCLC): {mean_hr:.1f}")

What the Python half tests: JSON parsing, careful regex on messy clinical free text — the NSCLC-vs-SCLC ordering trap is the whole point of Q1 and Q2 — and a guarded aggregation for the mean.

HCSC — Senior ML Engineer, live coding

Live coding over video call in a shared editor. SQL first, then Python. LeetCode-easy/medium difficulty, but they watch how you structure and explain. Narrate as you go — silence while typing is the failure mode, not wrong syntax.

SQL: Find the names of employees whose salary is higher than the average salary of departments with more than 3 employees.

employees(emp_id, name, department_id, salary)
departments(department_id, dept_name)

WITH dept_stats AS (
  SELECT department_id, AVG(salary) AS avg_salary
  FROM employees
  GROUP BY department_id
  HAVING COUNT(*) > 3
)
SELECT e.name
FROM employees e
JOIN dept_stats d ON e.department_id = d.department_id
WHERE e.salary > d.avg_salary;

Probed after: why HAVING and not WHERE, and the correlated-subquery alternative vs the CTE.

Python: valid parentheses. Given a string containing only ( ) { } [ ], determine if it's valid — brackets closed by the same type, in the correct order, every closer has an opener.

def is_valid(s):
    pairs = {')': '(', '}': '{', ']': '['}
    stack = []
    for ch in s:
        if ch in pairs:
            if not stack or stack.pop() != pairs[ch]:
                return False
        else:
            stack.append(ch)
    return not stack

O(n) time, O(n) space. Say the invariant out loud: the stack always holds the currently-unclosed openers, innermost on top.

Python: flatten nested JSON — nested keys join with underscores, so contact.phone.home becomes contact_phone_home.

def flatten(d, parent="", sep="_"):
    out = {}
    for k, v in d.items():
        key = f"{parent}{sep}{k}" if parent else k
        if isinstance(v, dict):
            out.update(flatten(v, key, sep))
        else:
            out[key] = v
    return out

The most DE-flavored Python question there is — it shows up everywhere. Know the recursive version cold, and expect the follow-up: what about lists inside the JSON? (Index into the key: contact_phones_0.)

What carries healthcare loops

Window functions (LAG/LEAD, ROW_NUMBER, rolling averages), the 30-day readmission pattern, SCD types, and honest depth on every tool your resume lists. Domain vocabulary helps too: HL7/FHIR, EHR data, HIPAA handling, claims vs clinical data.
