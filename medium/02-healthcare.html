Healthcare Data Engineering Interviews: Real Questions From Providence and HCSC

Field notes from two real healthcare data engineering loops (2026) — a Senior DE role at Providence and a Senior ML Engineer role at HCSC. Written down as asked, with clean solutions. Healthcare loops are practical — SQL, warehousing, pipelines — not algorithm puzzles, and they're winnable on fundamentals.

Providence — Senior Data Engineer

Typical loop: recruiter screen, structured behavioral (1 hour on video), coding/technical rounds, managerial, HR. Three to five weeks end to end.

The behavioral round, actually asked:

  • Tell me about a difficult situation and how you handled it.
  • Tell me about a time you helped a team thrive.
  • Why this role, why this organization?
  • How does your education align with this role?

Also expect questions pulled straight from your resume — even mid-behavioral, the interviewer may pick a listed tool (ADF, Synapse, Spark…) and ask you to walk through how you've used it. Know every line you wrote.

Azure Data Factory — enough to hold the conversation

  • The four objects: pipeline → activity → dataset → linked service. Be able to walk one flow through all four.
  • Integration Runtime: Azure IR (cloud) vs self-hosted IR. Healthcare loves self-hosted because clinical sources sit behind firewalls.
  • Triggers: schedule vs tumbling window — the distinction interviewers reach for. Tumbling windows are stateful, non-overlapping, and support backfill and dependencies.
  • The senior-sounding answer: metadata-driven pipelines — one parameterized pipeline plus a control table, instead of 200 copy-pasted pipelines.
  • Incremental loads: watermark column + lookup activity, idempotent re-runs.
  • Security (healthcare will care): managed identity over connection strings, Key Vault for secrets, private endpoints.
  • Comparisons to have ready: ADF vs Synapse pipelines, ADF vs Airflow, ADF vs Databricks workflows.

HCSC — Senior ML Engineer, live coding

Live coding over video call in a shared editor. SQL first, then Python. LeetCode-easy/medium difficulty, but they watch how you structure and explain. Narrate as you go — silence while typing is the failure mode, not wrong syntax.

SQL: Find the names of employees whose salary is higher than the average salary of departments with more than 3 employees.

employees(emp_id, name, department_id, salary)
departments(department_id, dept_name)

WITH dept_stats AS (
  SELECT department_id, AVG(salary) AS avg_salary
  FROM employees
  GROUP BY department_id
  HAVING COUNT(*) > 3
)
SELECT e.name
FROM employees e
JOIN dept_stats d ON e.department_id = d.department_id
WHERE e.salary > d.avg_salary;

Probed after: why HAVING and not WHERE, and the correlated-subquery alternative vs the CTE.

Python: valid parentheses. Given a string containing only ( ) { } [ ], determine if it's valid — brackets closed by the same type, in the correct order, every closer has an opener.

def is_valid(s):
    pairs = {')': '(', '}': '{', ']': '['}
    stack = []
    for ch in s:
        if ch in pairs:
            if not stack or stack.pop() != pairs[ch]:
                return False
        else:
            stack.append(ch)
    return not stack

O(n) time, O(n) space. Say the invariant out loud: the stack always holds the currently-unclosed openers, innermost on top.

Python: flatten nested JSON — nested keys join with underscores, so contact.phone.home becomes contact_phone_home.

def flatten(d, parent="", sep="_"):
    out = {}
    for k, v in d.items():
        key = f"{parent}{sep}{k}" if parent else k
        if isinstance(v, dict):
            out.update(flatten(v, key, sep))
        else:
            out[key] = v
    return out

The most DE-flavored Python question there is — it shows up everywhere. Know the recursive version cold, and expect the follow-up: what about lists inside the JSON? (Index into the key: contact_phones_0.)

What carries healthcare loops

Window functions (LAG/LEAD, ROW_NUMBER, rolling averages), the 30-day readmission pattern, SCD types, and honest depth on every tool your resume lists. Domain vocabulary helps too: HL7/FHIR, EHR data, HIPAA handling, claims vs clinical data.
