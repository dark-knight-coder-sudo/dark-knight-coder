A Disney Data Engineer Screen: Two Coding Problems and Two Questions You Answer on a Timer

Field notes from a real Data Engineer screen for a Disney role via a staffing firm (2026). Two parts: a timed coding assessment (stdin/stdout, multiple test cases per input), then a verbal technical round where you talk through scenarios on a roughly 3-minute clock. Everything below was actually asked.

Coding — Exponents of 2

Input: first line is T, the number of test cases. Each of the next T lines is a number. For each, print 1 if it is a power of two, else 0.

The catch: inputs can be non-integers, and the intended definition accepts fractional powers too (0.5, 0.25), since halving or doubling reaches exactly 1.0. Anything ≤ 0 is 0.

import sys

def is_power_of_two(n: float) -> bool:
    if n <= 0:
        return False
    while n > 1:
        n /= 2
    while n < 1:
        n *= 2
    return n == 1.0

data = sys.stdin.read().split('\n')
t = int(data[0])
out = ['1' if is_power_of_two(float(data[i])) else '0' for i in range(1, t + 1)]
print('\n'.join(out))

For integers only, the one-liner is n > 0 and (n & (n - 1)) == 0 — worth saying out loud even when the float version is what the tests require.

Coding — Sorting Bank Accounts

For each test case: an integer N, then N account-number strings. Print each unique account with its count, sorted by account number. A blank line separates test cases.

import sys
from collections import Counter

data = sys.stdin.read().split('\n')
idx = 0
t = int(data[idx].strip()); idx += 1
blocks = []
for _ in range(t):
    n = int(data[idx].strip()); idx += 1
    accounts = [data[idx + j].strip() for j in range(n)]
    idx += n
    if idx < len(data) and data[idx].strip() == '':   # skip separator
        idx += 1
    counts = Counter(accounts)
    blocks.append('\n'.join(f"{acct} {counts[acct]}" for acct in sorted(counts)))
print('\n\n'.join(blocks))

What it tests: careful index-based stdin parsing with a variable number of lines per case, Counter for the tally, and sorted() on the keys for the required ordering.

Verbal: the ETL data-quality question

"In an ETL process, what is a common method for handling data quality issues during the transformation phase?"

The answer that lands: validate on the way through and quarantine, don't drop. Schema/type checks and constraint rules (not-null, ranges, referential integrity) applied as the data transforms; failing rows routed to a dead-letter or reject table with the failure reason attached, instead of being silently discarded; standardization and dedupe; and data-quality metrics (rows in vs out, percent rejected, null rates) emitted so a regression is visible. Name a tool or two — dbt tests, Great Expectations, or hand-rolled rule tables.

Verbal: the Delta-table tuning question

"A 2-TB Delta table partitioned by event_date has slow reads and high cost. In 3 minutes, describe the exact sequence of Databricks actions you'd take to diagnose and improve performance and cost." (Hints offered: file sizes, small files, Z-ORDER targets, VACUUM/retention safety, Photon, caching, autoscaling, query-side techniques.)

A sequence that hits every hint:

  1. Diagnose first. Spark UI / query profile for the slow query, then DESCRIBE DETAIL for numFiles and total size — that gives average file size — plus a look at partition skew. Slow reads on a partitioned table are almost always the small-file problem.
  2. Compact. OPTIMIZE to coalesce small files toward the ~128 MB–1 GB target; enable auto-optimize / optimizeWrite so it stays compacted.
  3. Z-ORDER on the high-cardinality columns that actually appear in WHERE/join predicates — not the partition column. That's what turns full scans into data skipping.
  4. Reclaim safely. VACUUM old files but keep the default 7-day retention — never RETAIN 0 HOURS on a live table; it breaks time travel and concurrent readers.
  5. Engine and compute. Photon for vectorized execution; right-size the cluster with autoscaling; Delta caching for hot repeated reads.
  6. Query side. Push event_date predicates so partition pruning kicks in, select only needed columns, broadcast small dimension joins.
  7. Rethink the layout. If event_date produces tiny partitions, coarsen to month plus Z-ORDER — or move to liquid clustering.

The framing that scores: diagnose → compact → cluster → reclaim → tune compute → fix the query → rethink layout. Naming the file-size target and the VACUUM-retention safety note is what separates a senior answer from a keyword list.
